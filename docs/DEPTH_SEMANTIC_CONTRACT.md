# Depth Semantic Contract v1.0

> Required by deep-research-report §Media: *"Treat changes as breaking
> changes requiring full regression."*

## Overview

This document defines the **hard contract** for depth data flowing from
the RealSense D435 camera through WebRTC to viewers and consumers.
Any change to this contract is a **breaking change** and requires full
regression testing.

## Source: RealSense D435

| Parameter        | Value                         |
|------------------|-------------------------------|
| Sensor           | Intel RealSense D435 (D4xx)   |
| Depth format     | Z16 (16-bit unsigned, mm)     |
| Depth scale      | Read from `get_depth_scale()` |
| Color format     | RGB24 (8-bit per channel)     |
| Resolution       | 480 × 640 (rotated 90° CW)   |
| Frame rate       | 15 FPS (hardware-locked)      |
| Rotation         | 90° clockwise (`np.rot90`)    |

## Depth Encoding Mode: Pseudo-color Visualization (Option B)

The depth stream transmitted via WebRTC is a **pseudo-color
visualization** generated by `rs.colorizer()`.  It is suitable for
operator situational awareness but **NOT** for metric computation
by downstream consumers.

| Property                | Value                                      |
|-------------------------|--------------------------------------------|
| Encoding                | `rs.colorizer()` → RGB24 → H264 (lossy)   |
| Semantic content        | **Visual-only** — pixel values ≠ distance  |
| Range mapping           | Colorizer uses jet palette, near=red far=blue |
| Invalid depth (Z16 = 0) | Rendered as black (RGB 0,0,0)             |
| Out-of-range            | Rendered as extreme end of palette         |
| Quantization loss       | H264 compression destroys exact colors     |

### Raw Depth Access (metric)

For **metric depth queries**, use the HTTP API on the depth node:

```
GET http://192.168.1.55:8000/depth?x=50&y=50
→ {"depth_m": 1.234, "x_norm": 50, "y_norm": 50}
```

| Property           | Value                                   |
|--------------------|-----------------------------------------|
| Data type          | float32 (meters)                        |
| Conversion         | `z16_value * depth_scale` → meters      |
| Coordinate system  | Image plane, **post 90° CW rotation**   |
| Invalid depth      | 0.0 meters (Z16 source was 0)          |
| Max reliable range | ~10 m (D435 spec)                       |

Full depth map:
```
GET http://192.168.1.55:8000/depth_map?format=raw
→ binary float32 array, shape (480, 640), row-major
```

## Timestamp Policy

| Clock             | Source                                    |
|-------------------|-------------------------------------------|
| Frame timestamp   | `time.time()` at frame capture (Python)   |
| Authoritative     | Linux system clock on depth node          |
| NTP sync          | Yes (verified via timedatectl)            |
| Hardware sync     | Color↔depth: same FPS (15), but **no      |
|                   | hardware-level per-frame alignment**      |
| Discontinuities   | On pipeline restart, timestamp resets     |

**Warning**: Color and depth frames are from the same RealSense pipeline
call (`pipeline.wait_for_frames()`) but `rs.align()` is NOT used.
Spatial registration between color and depth is approximate.

## Breaking Change Checklist

Any modification to the following requires full regression:

- [ ] Resolution or FPS change
- [ ] Colorizer palette or settings
- [ ] Rotation direction or angle
- [ ] Depth scale source or conversion formula
- [ ] `rs.align()` addition/removal
- [ ] H264 encoder profile or settings
- [ ] Raw depth API response format
- [ ] Camera intrinsic values
